hang# 安得家政 CRM 项目整改方案与执行清单

## 1. 目标

本文档用于统一梳理当前项目在架构、权限、安全、业务一致性和小程序兼容性方面的主要问题，并给出分阶段整改方案。

执行原则：

- 先出方案，再改代码。
- 先做低风险、低影响的收敛项。
- 所有可能影响小程序或现网业务的改动，必须先走兼容方案。
- 不做一次性硬切，优先采用双轨、兼容层、开关、灰度和可回滚方案。
- 每完成一项，在状态文档中更新状态。

## 2. 项目现状概览

### 2.1 仓库形态

当前仓库是单仓库、多业务域并存的形态，核心包含：

- CRM 后台前端：`frontend`
- NestJS 后端：`backend`
- 小程序配套 API：客户、合同、简历、保险、背调等多个独立入口
- 第三方集成：爱签、大树保、ZEGO、微信、小程序用户、背调
- 旧版 H5、小程序页面、SDK、备份、构建产物、历史文档

### 2.2 当前主要风险特征

- 管理端 API 与小程序 API 边界不够清晰
- 资源级权限控制不一致
- 公开接口和测试接口混杂
- DTO 校验策略过于宽松
- 为兼容历史数据保留了大量 `temp`、宽松字段、手工兜底逻辑
- 小程序已深度依赖部分历史接口行为，不能直接硬收口
- 高风险主链路测试覆盖不足

## 3. 整改分级原则

### P0

必须优先处理，存在安全、越权、伪造、敏感数据泄露、直接事故风险。

### P1

高风险但需要兼容迁移，涉及现网业务链路或小程序依赖，不能硬切。

### P2

中期治理项，解决架构腐化、工程可维护性和性能问题。

### P3

优化项，影响维护效率、排障效率和长期扩展性。

## 5. 小程序影响评估

### 5.1 可以直接整改、通常不应影响小程序的项

- R01 爱签回调验签收紧
- R02 清除登录限制公开接口下线
- R03 CRM 客户详情资源级鉴权
- R04 CRM 合同详情资源级鉴权

- R16 仓库清理

### 5.2 需要兼容迁移、不能直接硬改的项

- R05 合同公开查询接口收口
- R08 全局 DTO 校验收紧
- R09 `temp` 占位逻辑清理
- R10 API 分层
- R11 小程序守卫统一
- R15 合同/爱签/保险链路改造

### 5.3 当前小程序依赖较深的接口类型

经检索，以下类型需要重点兼容：

- `/api/customers/miniprogram/*`
- `/api/contracts/miniprogram/*`
- `/api/dashubao/miniprogram/*`
- `/api/resumes/miniprogram/*`
- `/api/auth/miniprogram-login`
- `/api/miniprogram-users/*`

## 6. 推荐整改顺序

### 阶段 A：只做安全收口，不改协议

目标：降低直接事故风险，不改变小程序和 CRM 正常入参格式。

包含任务：

- A1 修复爱签回调未配置公钥仍放行
- A2 下线 `clear-login-limit`
- A3 清理明显测试接口和调试接口，先做使用情况核查
- A4 仅对 CRM 管理端详情接口增加资源级鉴权
- A5 为高风险公开接口增加访问日志和调用埋点

预期影响：

- 对小程序协议基本无影响
- 对不合规测试调用会有影响

### 阶段 B：补清单、补边界、做兼容分流

目标：搞清楚谁在调用、哪些接口能收口、哪些必须保留兼容层。

包含任务：

- B1 建立“管理端 API / 小程序 API / 公开 API / 内部回调 API”分层清单
- B2 标记所有 `@Public()` 接口用途、调用方、下线风险
- B3 标记所有测试接口、调试接口、历史兼容接口
- B4 对公开查询型接口补“替代接口”
- B5 为迁移项定义 v2 或新路径

预期影响：

- 无直接业务影响
- 为后续兼容迁移提供依据

### 阶段 C：兼容迁移

目标：在不硬切现网的前提下收紧危险行为。

包含任务：

- C1 将小程序仍在使用的历史接口迁到 `miniprogram` 专用路径
- C2 旧接口保留，但增加弃用标识、日志、限流或更严格鉴权
- C3 收紧 DTO 校验，但按模块推进，不做一次性全局切换
- C4 清理 `temp` 占位链路，先记录触发点，再替换
- C5 统一小程序守卫和身份注入行为

预期影响：

- 对小程序有一定适配成本
- 必须先联调、再切换

### 阶段 D：结构治理

目标：降低持续维护成本和后续变更风险。

包含任务：

- D1 路由单源化
- D2 模块边界梳理
- D3 仓库清理
- D4 测试补齐

## 7. 执行策略

### 7.1 代码变更原则

- 先补文档和状态，不直接改行为。
- 每次只改一类问题，不跨多个业务域同时推进。
- 高风险改动先做日志和埋点。
- 对兼容项优先新增而不是替换。
- 对核心流程不大量保留注释掉的死代码，优先保留兼容分支、开关或替代接口。

### 7.2 兼容策略

- 旧接口保留一段时间
- 新接口先上线
- 前端/小程序切换后观察调用日志
- 确认旧接口无调用再下线

### 7.3 回滚策略

- 所有整改按模块提交
- 每项整改保持独立可回退
- 高风险改动前先记录当前行为和关键样例

## 11. 爱签回调链路核查结论

### 11.1 当前链路

当前爱签回调入口为：

- [esign.controller.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/esign.controller.ts#L1192)

当前回调处理流程：

1. 接收 `/api/esign/callback`
2. 直接调用 `esignService.handleContractCallback(callbackData)`
3. 根据回调中的 `contractNo/status/account/signMark/signUserList` 更新通知文案
4. 触发微信通知
5. 当 `status=2` 时触发保险同步
6. 最终无论成功失败都返回 `"ok"`

### 11.2 当前主要问题

| 编号 | 问题 | 说明 |
|---|---|---|
| E01 | 控制器入口未见签名校验调用 | 当前回调入口直接处理业务数据，没有在控制器层读取签名头并校验 |
| E02 | 回调校验服务在“无公钥”时默认放行 | 见 [esign-callback.service.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/services/esign-callback.service.ts#L29) |
| E03 | 爱签公钥存在默认值 | 见 [esign-api.service.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/services/esign-api.service.ts#L39) |
| E04 | 爱签私钥也存在默认兜底 | 私钥缺失时会落到内置默认私钥，不利于环境显式配置 |
| E05 | 回调失败仍返回 `ok` | 这有利于避免对方重试，但会掩盖真实失败，需要更强日志和告警 |

### 11.3 对签约业务的影响判断

当前如果直接把“未配置公钥也放行”改成“未配置公钥即拒绝”，存在业务影响风险。

原因：

- 回调入口是合同状态同步、客户状态同步、通知、保险同步的关键节点
- 当前配置服务中，公钥与私钥都带默认值，不代表生产配置一定正确，只代表代码不会报缺配置
- 在未确认现网环境回调签名头格式、实际配置与最近成功回调情况之前，不适合直接收紧

### 11.4 爱签回调整改建议顺序

建议按以下顺序推进，而不是直接切拒绝：

1. 先补文档和日志
2. 核查现网环境是否真实配置 `ESIGN_PUBLIC_KEY`、`ESIGN_PRIVATE_KEY`、`ESIGN_NOTIFY_URL`
3. 核查最近回调是否确实正常落合同状态
4. 在控制器层补签名头读取与验签日志，不先改变结果
5. 验证一段时间后，再将“未配置公钥放行”切换为“拒绝”

### 11.5 爱签回调相关配置点

见：

- [esign-api.service.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/services/esign-api.service.ts#L39)
- [esign-callback.service.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/services/esign-callback.service.ts#L29)
- [esign.controller.ts](/home/ubuntu/andejiazhengcrm/backend/src/modules/esign/esign.controller.ts#L1192)

### 11.6 当前结论

当前不建议立即修改“未配置公钥即拒绝”。

当前最稳妥的动作应是：

- 先把爱签回调列为重点核查链路
- 先确认实际配置和真实回调行为
- 再决定何时收紧

### 12.6 结论

四条主链中，当前最不能硬动的是：

1. 合同
2. 简历
3. 保险

相对最稳的是：

1. 客户链路的文档与守卫核查
2. 小程序用户登录注册链路的日志补强

因此，后续如果要继续做“最小影响整改”，建议优先级为：

1. 先动不影响小程序协议的测试/临时接口
2. 再动只影响 CRM 的资源级鉴权
3. 小程序链路继续补文档、日志、依赖统计
4. 最后才动合同/简历/保险的兼容迁移
